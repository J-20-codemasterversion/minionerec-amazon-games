# 推荐系统 Demo 搭建全流程

## 一、整体规划

```
数据准备 → 召回 → 粗排 → 精排 → 重排 → 评估 → 部署演示
```

本项目基于 Amazon 5-core 评论数据（Video_Games + Electronics），搭建一套完整的四阶段推荐系统 Demo。以下逐步说明每一步做什么、用什么模型、得到什么结果。

---

## 二、Step 1：数据准备

### 做什么

从 Amazon 分品类 5-core 数据集中加载评论和商品元数据，选取活跃用户，构建交互矩阵。

### 具体操作

| 步骤 | 操作 | 产出 |
|------|------|------|
| 加载评论 | 读取 `Video_Games.json`（250万条）+ `Electronics.json`（采样300万条） | 全量评论池 |
| 选取用户 | 按交互数排序，选 top 10,000 活跃用户（每人 ≥ 10 条交互） | 种子用户集 |
| 截取交互 | 每用户按时间排序，保留前 100 条 | 276,833 条评论 |
| 加载 Meta | 从 `meta_Video_Games.json` + `meta_Electronics.json` 加载商品元数据 | 47,160 个商品的 title/brand/category/price/also_buy/also_view |
| 构建索引 | `uid2idx`/`iid2idx` 双向映射、`user2items`/`item2users` 交互集合 | 交互矩阵 |
| 统计特征 | 用户平均评分/评分次数/评分标准差、商品平均评分/评价次数、全局均值 | 基础统计特征 |
| 数据划分 | Leave-Last-One-Out：每用户最后 1 条交互做测试，其余做训练 | 训练集 + 测试集 |

### 得到的结果

```
用户数: 10,000
商品数: 47,160
总评论: 276,833
平均交互/用户: 27.7
平均交互/商品: 5.9
Meta 覆盖率: ~95%
加载耗时: ~26 秒
```

### 为什么这么做

- **5-core 保证**：每个用户和商品至少 5 条交互，协同过滤有足够信号
- **不用全量数据**：全量 All_Amazon 有 2.3 亿行（82GB），扫描前 4% 会导致大量用户交互被截断（平均只剩 1.9 条），破坏数据质量
- **分品类子集**：数据完整、Meta 配套、also_buy/also_view 保留率高

## 三、Step 2：多路召回（47,160 → 1,000）

### 做什么

从 47,160 个商品中，为每个用户快速筛出 1,000 个"可能感兴趣"的候选。

### 模型选择：7 路召回 + 融合

| 通道 | 模型/方法 | 权重 | 核心思想 |
|------|-----------|------|----------|
| Hot 热门 | 按评价数排序 | 0.10 | 大家都买的你可能也想买（兜底） |
| ItemCF | 协同过滤（商品相似度矩阵） | 0.30 | 买了 A 的人也买了 B |
| UserCF | 协同过滤（用户相似度矩阵） | 0.20 | 和你相似的人还买了什么 |
| Swing | 改进版 ItemCF | 0.40 | 小圈子共现比大众共现更有价值 |
| Content | 品牌/品类/标题关键词倒排索引 | 0.25 | 同品牌/同品类的商品 |
| Also-Buy/View | Amazon 预计算的商品关联图谱 | 0.50 | Amazon 全量用户行为提炼的关联 |
| 品类热门 | 用户偏好品类的热门商品 | 0.15 | 在你喜欢的品类里推热门 |

### 融合策略

- 各通道分数归一化后加权合并
- 被多个通道同时召回的商品额外加分：`score × (1 + 0.15 × 命中通道数)`
- 取 Top-1000

### 得到的结果

```
Recall@1000 = 28.00%（即 28% 的用户的正确答案在这 1000 个候选里）
各通道可达性：
  content_cat: 47.5%
  usercf: 32.0%
  content_brand: 32.0%
  itemcf: 31.5%
  swing: 31.0%
  also_buy: 21.5%
  also_view: 16.5%
```

## 四、Step 3：粗排（1,000 → 100）

### 做什么

用机器学习模型对 1,000 个候选快速打分，保留最可能被用户交互的 Top-100。

### 模型选择：XGBoost 二分类

**为什么用 XGBoost**：
- 树模型自动处理特征交叉，不需要手动构造
- C++ 多线程实现，对 1,000 个候选批量打分极快
- 内置正则化（L1/L2/max_depth/min_child_weight），不容易过拟合

### 特征设计（6 维轻量特征）

| 特征 | 含义 | 为什么用 |
|------|------|----------|
| co_interact | 用户邻居中买过该商品的人数 | 协同过滤信号 |
| jaccard | 用户商品集与候选商品购买者商品集的 Jaccard 系数 | 兴趣重叠度 |
| item_rating_cnt | 商品被评价次数 | 商品热门度 |
| item_avg_rating | 商品平均评分 | 商品质量 |
| user_avg_rating | 用户平均评分 | 用户评分倾向 |
| item_pop_rank | 商品热门度排名（log变换） | 相对流行度 |

**为什么只用 6 维**：粗排要快，只用最有区分力的特征。不含召回分数（避免训练泄露）。

### 训练方式

```
正样本 = 用户真实交互的商品（label=1）
负样本 = 每个正样本配 4 个随机未交互商品（label=0）
总样本 = ~250,000（50K 正 + 200K 负）
训练/验证 = 80%/20%
Early stopping: 验证集 15 轮不提升则停止
```

### 推理

```python
最终分数 = XGBoost预测概率 + 0.3 × 归一化召回分数
```

### XGBoost 关键超参数

| 参数 | 值 | 作用 |
|------|-----|------|
| max_depth | 5 | 每棵树最多 5 层，防过拟合 |
| learning_rate | 0.05 | 每棵树只贡献 5% 修正，慢慢学 |
| n_estimators | 500 | 最多 500 棵树（early stopping 通常 ~167 棵） |
| subsample | 0.8 | 每棵树随机用 80% 样本 |
| colsample_bytree | 0.8 | 每棵树随机用 80% 特征 |
| min_child_weight | 50 | 叶子节点至少 50 个样本 |

### 得到的结果

```
粗排 HitRate@10 = 4.00%（与召回一致，无损失）
粗排 AUC = 0.5222
训练 AUC ≈ 验证 AUC（无过拟合）
```

## 五、Step 4：精排（100 → 30）

### 做什么

对粗排筛出的 100 个候选做精确排序，用更丰富的特征和更强的模型区分"看起来都差不多"的候选。

### 模型选择：FM + BPR（因子分解机 + 贝叶斯个性化排序）

**为什么用 FM**：
- 通过隐向量自动学习任意两个特征之间的交叉关系（如"喜欢 Sony 的用户 × 价格 50-100 的商品"）
- 比 XGBoost 更适合排序任务（配合 BPR pairwise loss）
- 参数量可控，不需要 GPU

**为什么用 BPR 而非二分类**：
- 粗排做二分类："这个商品用户会不会买"（pointwise）
- 精排做排序："A 和 B 哪个更值得推荐"（pairwise）
- 排序更贴合实际需求——我们不关心绝对概率，只关心相对顺序

### 特征设计（16 维，在粗排基础上扩展）

| 类别 | 特征 | 维度 | 新增说明 |
|------|------|------|----------|
| 用户特征 | avg_rating, rating_cnt, rating_std | 3 | 同粗排 |
| 商品特征 | price, log_rank, n_also_buy, n_also_view, avg_rating, rating_cnt | 6 | 新增价格、排名、关联数量 |
| 交叉特征 | co_interact, jaccard, cat_pref, brand_pref | 4 | 新增品类/品牌偏好匹配 |
| 召回分数 | itemcf_score, swing_score, usercf_score | 3 | 新增各通道真实召回分数 |

### 训练方式

```
训练数据 = (user, pos_item, neg_item) 三元组，共 200,000 对
目标: score(user, pos) > score(user, neg)
Loss = -log σ(s_pos - s_neg)    ← BPR Loss
优化器: Mini-batch SGD, batch=512
Epochs: 最多 30，early stopping patience=5
学习率: 0.005
正则化: L2 (w=0.01, v=0.01)
梯度裁剪: ±5.0
FM 隐向量维度: K=8
```

### FM 模型结构

```
score = w0 + w_user[u] + w_item[i] + w · x
        + 交互项: 0.5 × [||v_u + v_i + V·x||² - (v_u² + v_i² + V²·x²)]
```

- `w0`: 全局偏置
- `w_user[u]`, `w_item[i]`: 用户/商品一阶偏置
- `w · x`: 16 维特征的线性加权
- 交互项: 自动学习特征两两交叉

### 推理

```python
最终分数 = FM排序分 + 0.3 × 粗排分数
```

### 得到的结果

```
精排 HitRate@10 = 2.00%
精排 AUC = 0.5053
BPR 验证 AUC = 0.9992（训练/验证一致，无过拟合）
```

## 六、Step 5：重排（30 → 10）

### 做什么

从精排的 30 个候选中挑出最终 10 个推荐给用户，同时保证推荐列表的多样性。

### 模型选择：无模型，贪心规则策略

**为什么不用模型**：重排的目标不是"更准"而是"更好用"——避免推荐 10 个同品牌商品、给冷门好物曝光机会、奖励高置信候选。这些是业务规则，不需要学习。

### 策略

逐个贪心选择，每选一个后重新计算剩余候选的分数：

```
final_score = FM精排分数 + 品牌去重惩罚 + 新鲜度奖励 + 多源召回奖励
```

| 规则 | 条件 | 加分/扣分 |
|------|------|-----------|
| 品牌去重 | 该品牌已在推荐列表中出现过 | **-0.3** |
| 新鲜度 | 商品评价数 < 10（冷门/新品） | **+0.05** |
| 多源召回 | 被 N 个召回通道同时命中 | **+0.05 × N** |

### 得到的结果

```
重排 HitRate@10 = 2.00%
重排 AUC = 0.5061
平均品牌多样性（unique/total）≈ 0.8+
```

## 七、Step 6：评估与验证

### 评估方法

- **Leave-Last-One-Out**：每用户最后 1 条交互做测试
- **评估样本**：200 个有效测试用户
- **在每个阶段独立计算指标**

### 评估指标

| 指标 | 含义 |
|------|------|
| Recall@1000 | 召回池中包含正确答案的用户占比 |
| HitRate@10 | 最终 Top-10 推荐中包含正确答案的用户占比 |
| AUC | 正样本排在负样本前面的概率 |

### 各阶段指标对比

| 阶段 | 输入→输出 | HitRate@10 | AUC |
|------|-----------|------------|-----|
| 召回 | 47,160 → 1,000 | 4.00% | — |
| 粗排（XGBoost） | 1,000 → 100 | 4.00% | 0.5222 |
| 精排（FM-BPR） | 100 → 30 | 2.00% | 0.5053 |
| 重排（规则） | 30 → 10 | 2.00% | 0.5061 |

### 阶段衰减分析

```
Recall@1000 = 28.00%（候选池的天花板）
召回 → 粗排 HitRate 保留率: 100%（粗排未丢命中）
粗排 → 精排 HitRate 保留率: 50%
精排 → 重排 HitRate 保留率: 100%
```

### 健康检查项

- 数据：交互密度、平均交互数、评分分布、Meta 覆盖率
- 召回：Recall@1000、空召回率、各通道贡献占比
- 粗排：训练/验证 AUC 一致性、特征重要性分布
- 精排：BPR Loss 下降趋势、权重是否有 NaN/Inf
- 重排：品牌多样性得分

## 九、模型选择总结

| 阶段 | 模型 | 为什么选它 |
|------|------|-----------|
| 召回 | ItemCF / UserCF / Swing / Content / Also-Buy | 经典协同过滤 + 内容 + 知识图谱，多路互补覆盖面广 |

## 十、已知瓶颈与改进方向

### 当前瓶颈

1. **召回天花板**：Recall@1000 = 28%，即 72% 的正确答案根本不在候选池里，后续排序再好也无法挽回
2. **精排 AUC ≈ 0.5**：在粗排筛出的 Top-100 中，精排几乎无法区分正负样本——当前 16 维特征在"精细区分"场景下信号不足

### 改进方向

| 方向 | 做法 | 预期收益 |
|------|------|----------|
| 扩大召回量 | Top-K 从 1000 提到 2000-3000 | Recall 可能 +5-10% |
| 向量召回 | Item2Vec / Word2Vec embedding | 覆盖长尾商品 |
| 更强精排模型 | LightGBM / DeepFM 替代手写 FM | 更好的特征交叉能力 |
| 更丰富的特征 | 评论文本 TF-IDF、价格区间偏好、品类交叉统计 | 给精排更多区分信号 |
| Hard Negative | 从召回列表中采负样本而非全局随机 | 训练/评估分布对齐 |
